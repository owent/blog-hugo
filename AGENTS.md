# blog-hugo AI 协作指南

## 事实源与工作方式

- 用户当前指令优先；目录内存在更近的 `AGENTS.md` 时，再叠加其局部规则。
- 动手前只读取与任务有关的入口、目标文件、相邻资源和必要参考，不要预先展开整个仓库或全部 Skill。
- 先核对源码、配置、构建结果和已有文章。涉及模型、版本、价格、兼容性、规范或其他易变事实时，查当前官方资料或一手来源。
- 区分已验证事实、合理推断和未验证信息。不能确认的关键结论继续调研；仍无法确认时明确标注，不要补写成确定事实。
- 采用最小充分改动，保留用户已有内容和工作区中的无关修改。验证用户要求的目标，不用相邻检查代替。

## 仓库约束

- Hugo 内容源在 `source/`，生成结果在 `public/`。除非用户明确要求，不直接编辑 `public/`。
- 文章位于 `source/post/<year>/<id>.md`。`config.yaml` 使用
  `permalinks.post: /:year/:contentbasename` 和 `uglyURLs: true`，因此现有年份目录、数字文件名和 `id` 都属于稳定链接的一部分。
- 修改已有文章时，不改文件名、年份目录、`id` 或稳定锚点；同时保留有意使用的 front matter、引用、脚注和资源名。
- 新增文章资源放在同一年目录，文件名使用 `<id>-<description>.<ext>`。

## 任务分流

| 任务 | Skill |
| --- | --- |
| 新写文章、把笔记整理成新稿、重写为一篇新文章 | `article-writer` |
| 润色、扩写、校对或修复已有文章 | `article-refiner` |

先读匹配的 `.agents/skills/<name>/SKILL.md`；只有任务需要图表嵌入细节时，再读该 Skill 的 `references/embedding-syntax.md`。主题、样式、脚本或普通仓库维护不触发文章 Skill。

## 写作基线

- 默认使用简体中文；用户明确要求英文或双语时再切换。
- 默认读者是有经验的软件工程师。写清问题、证据、约束、权衡、适用条件和验证方法，删掉没有信息量的正确废话。
- 保留作者已有的第一人称、判断、怀疑、取舍和适度口语。不要把工程复盘磨成中性报告，也不要凭空补作者经历、情绪或观点。
- 开头尽快进入实际问题，并自然说明读者为什么值得继续读。收益和边界只写与本文有关的内容，不机械套用固定开场模板。
- 结构跟着论证走。需要做判断或选型时可以结论前置；学习笔记、推导和排障复盘可以保留必要的过程顺序。
- H2/H3 对应真实的主题转折。不要为了形式完整把相近原因拆成多个小节，也不要让每节都使用相同的“结论—解释—总结”节奏。
- 段落保持易读，但不硬性追求同样长度。列表、表格只用于并列项、步骤、字段映射和比较；连续论述优先使用自然段。
- 代码或图表前说明它解决什么问题。只保留支撑当前论点的内容；数据是结论依据时，保留原始表格和关键数值。
- 结论依赖版本、平台、编译器、网络环境、时间点或实验条件时，紧邻结论写出前提。

### 避免模板化的 AI 文风

- 优先使用具体名词、动作、数据和因果关系。删减“本文将”“值得注意的是”“综上所述”这类不推进内容的元话语。
- 检查整篇的重复模式：连续对偶句、机械三段式、相同长度的小节、每段末尾的小结、过密转场词，以及为完整感重复上一段结论。
- “不是 A，而是 B”“不只是 A”“从 X 到 Y”“更稳”“更安全”“更像”“守住边界”“收敛输出”等只是高频症状，不是禁词。只有在密集、空泛或可直接改写时才处理；正常表达不必为了避词而扭曲。
- 把抽象口号换成工程对象，例如拒答条件、偏好数据、评估集、长尾 case、延迟、成本或版本前提。
- 样例代码、粗略数字和占位路径通常只需一句边界说明。不要把一句提醒扩写成完整免责声明。
- 比喻只在确实降低理解成本时使用；能用准确术语和具体场景说明时直接写。
- 不要自动弱化作者的负面判断或口语化风险表达。先确认它有事实或上下文支撑，再决定保留、收紧或标注为个人判断。
- 润色“AI 味”时先找同类模式，再做段落级和全文级调整。不要只做词语替换，也不要把句长变化、口语和第一人称机械注入每一段。
- 禁止通过故意错字、语病、伪造引语、虚构经历或规避 AI 检测器来制造“人味”。目标是保留作者声音并提高信息密度，不是伪装作者身份。
- 需要校准作者声音时，按主题抽样阅读 2 至 3 篇作者已有文章；不要预读全库，也不要照抄固定口头禅。

## 新写与改文

### 新写文章

- 先确认主题、目标读者、核心判断、已有材料和不能编造的缺口。信息不足以支撑正文时，先给缺口和结构建议。
- 搜索相近文章，避免重复背景；需要时用内部链接建立上下文。
- 根据材料选择结构，不强制套用统一章节模板。先写有证据的部分，再补过渡和摘要。

### 修改已有文章

- 先完整读取 front matter、正文、引用、脚注和关联资源，再判断需要事实修复、结构调整还是轻量措辞优化。
- 优先级依次为：事实与术语错误、论证和结构问题、缺失前提与验证信息、措辞与节奏。
- 保持 diff 克制。用户要求直接修改仓库文件时，完成修改和验证，不只给审阅报告。
- 用户指出某类模板腔或过度解释时，扫描全文同类模式；保留作者核心立场、数据证据和有意留下的内容。

## Hugo 与 Markdown

- 文章 front matter 必须是合法 YAML，并至少保留 `title`、`date` 和 `type`；发布状态、`id`、标签、分类和作者字段按文章现状或用户要求处理。
- 所有围栏代码块都标注语言；日志和纯文本使用 `text`。
- 本站文章链接最终为 `/:year/:contentbasename.html`。新增内部链接前核对实际生成路径，不沿用未经验证的示例。
- 中英文、数字与中文之间保留合理空格；术语、大小写和标点在全文保持一致。

### 图表

- 只有当纯文字会迫使读者脑内模拟流程、时序、状态、组件关系、拓扑或数据变化时才加图。
- Mermaid 用于流程、时序和状态；Chart.js 用于简单数值图；ECharts/Plotly 用于确有必要的复杂交互或科学图；Excalidraw 用于解释型草图；Draw.io 用于正式且几何关系稳定的图。
- 简短图表可内嵌；较长、可复用或参数复杂的图拆成外部文件。文件型图表要有有意义的 alt 文本和 1 至 2 句图注。
- 修改内联 Mermaid 时逐行核对并实际验证渲染；复杂图优先使用外部资源，避免留下“看起来能解析”但未经验证的图。

## 交付前验证

- 核对标题、摘要和正文是否表达同一结论，开头是否直接进入问题。
- 核对事实、引用、版本前提、术语和代码示例；不要编造 benchmark、Issue、官方结论、支持矩阵或用户经历。
- 做一次全文模式检查，重点看模板化转场、重复总结、均匀小节、抽象口号、长免责声明和被抹平的作者判断。
- 检查 YAML、围栏语言、资源命名、内部链接和图表说明。
- 修改文章后运行 `hugo --buildDrafts --renderToMemory`。若本地可用 markdownlint-cli2，按 `.markdownlint-cli2.yaml` 检查改动文章；工具不可用时说明验证边界。
- 修改图表时运行对应渲染器或语法检查。修改 SCSS 时按下文重新生成 CSS。
- 最后运行 `git diff --check`，复查 diff 中是否有无关改动、临时文件或生成物。

## AI 提示词与 Skill 维护

- `AGENTS.md` 是全局规则唯一主源。`.claude/CLAUDE.md` 只导入它并保留 Claude 专属差异；`.github/instructions/` 只保留路径触发所需的最小提示。
- `.agents/skills/` 是 Skill 主副本，`.claude/skills/` 是兼容镜像。只编辑主副本，再运行 `sync-skills.ps1`。
- 全局写作规则只写在本文件。Skill 只保留触发边界、任务独有流程、按需参考和验收差异。
- Skill `description` 说明用户意图、适用场景和相邻 Skill 边界；正文只写 Agent 容易做错且无法从本文件或仓库推断的内容。
- 修改提示词或 Skill 时先删重复、过时、低信号和可由源码推断的规则。不要在常驻提示里保留版本历史、维护日期或调研过程。
- 外部规范和研究依据放在 `docs/ai/source-index.md`，按需读取。正文只保留当前可执行结论，不复制来源摘要。
- 修改 Skill 后运行可用的规范校验器，检查名称、目录、front matter 和描述长度；再同步镜像并核对哈希。

## 主题与样式开发

- 主题源码位于 `themes/distinctionpp/`。
- SCSS 入口是 `themes/distinctionpp/static/css/style.scss`；不要直接编辑生成的 `style.css`。
- 修改 SCSS 后在 `themes/distinctionpp` 运行：

  ~~~bash
  sass -s compressed --source-map -c static/css/style.scss static/css/style.css
  ~~~

- 首页和归档页文章列表使用 CSS Grid；默认单列，`576px` 以上切换为响应式多列。归档卡片使用固定高度和 `-webkit-line-clamp` 保持网格整齐。

## 禁止事项

- 不编造数据、来源、官方结论、用户经历或作者观点。
- 不擅自更改已有文章的稳定路径要素。
- 不在信息不足时强行生成完整文章。
- 不为展示工作量而过度改写、过度加图或增加空文档。
- 不删除作者有意保留的内容，不代替用户做技术选型决策。
