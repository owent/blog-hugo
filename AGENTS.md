# blog-hugo agent guide

## 项目定位

- 这是一个 Hugo 技术博客仓库，主要内容源文件位于 `source/`，最终生成目录为 `public/`。
- 文章正文主要位于 `source/post/<year>/<id>.md`。
- `config.yaml` 当前启用了 `defaultContentLanguage: zh-cn`、`permalinks.post: /:year/:contentbasename`、`uglyURLs: true`。
- 这意味着文章的年份目录、数字文件名和 `id` 会直接影响最终链接；除非用户明确要求，否则不要改已有文章的编号、年份目录或文件名。

## 写作默认值

- 默认输出语言使用简体中文；仅在用户明确要求英文或双语时切换。
- 目标读者通常是有经验的软件工程师；优先写清问题、约束、权衡、落地方法，少空话、少营销腔。
- 开头前 1-3 段要交代：问题是什么、为什么值得读、读者读完能获得什么、本文不打算覆盖什么。
- 结论前置：每节优先给结论或适用条件，再解释原因，再给示例、代码或图。
- 长文优先使用清晰、描述性的 H2/H3 标题；让只扫标题的读者也能判断哪些段落和自己相关。
- 段落尽量短；流程、枚举、对比、决策条件优先用列表或表格表达，而不是堆成长段落。
- 代码示例前先说明它证明什么或解决什么；只展示和当前论点直接相关的片段。
- 如果某个结论依赖版本、平台、编译器、网络环境、时间点或实验条件，必须显式写出前提。
- 不要编造 benchmark、Issue 编号、官方结论、版本支持矩阵、引用来源或用户未提供的经历细节；不确定时明确写“待确认”或“需要验证”。

## Hugo / Markdown 约定

- 始终保持合法 YAML front matter。文章常见字段包括 `title`、`id`、`date`、`draft`、`type: post`、`tags`、`categories`、`author`。
- 修改已有文章时，优先保留现有 front matter、引用链接、脚注编号、资源文件名和整体章节骨架，只做必要改动。
- 新文章若无其他要求，可沿用：`type: post`、`author: owent`，并给出克制且有检索价值的 `tags` 与 `categories`。
- 代码块、Mermaid、KaTeX、Draw.io、Excalidraw、Chart.js 仅在确实能降低理解成本时使用；不要为了“看起来更专业”而机械加图。
- 如果纯文字会迫使读者脑内模拟流程、状态机、调用时序、组件关系、跨区拓扑或数据变化，AI 应主动在对应段落附近补图，而不是等用户点名。
- 图表选型遵循以下默认规则：
	- **Mermaid**：适合流程图、时序图、状态图、类图、ER 图、甘特图、象限图、简单文本驱动架构图；优先用于可直接用文本维护的结构化关系。
	- **Chart.js**：适合有数值的比较、趋势、占比、分布、混合折线/柱状对比；优先用于“看数据关系”而不是“看系统结构”。
	- **Excalidraw**：适合概念草图、粗粒度架构、角色/组件关系、需要“白板感”的解释型图示；优先用于帮助读者建立心智模型。
	- **Draw.io**：适合布局更规整、节点更多、需要稳定几何关系或更正式观感的架构/网络/部署图。
- 这个仓库支持 4 种嵌入方式：图片语法引用外部文件、代码块内嵌、shortcode 引用外部文件、shortcode 内联；默认写法优先级为：
	- 简短 **Mermaid / Chart.js**：优先代码块（`mermaid`、`chart`、`chartjs`）。
	- 较长或可复用的 **Mermaid / Chart.js**：优先拆成外部 `*.mermaid` / `*.chart.json` 文件。
	- 非平凡 **Excalidraw / Draw.io**：优先外部 `*.excalidraw` / `*.drawio` 文件，并在正文中用图片语法引用。
	- Shortcode `src` / inline 两种方式都支持，但默认不要优先选它们，除非代码块或文件引用明显更别扭。
- 新增配套资源时，请放在同一年目录下，并与文章编号保持同一前缀，例如 `2604-*.mermaid`、`2604-*.drawio`、`2604-*.excalidraw`、`2604-*.chart.json`。
- 文件型图表的图片语法要写有意义的 alt 文本；正文中也要用 1-2 句说明“这张图要帮助读者看什么”，不要只丢一个图让读者自己悟。

## 改文策略

- 写新文时，先补齐主题、目标读者、核心结论和章节结构，再写正文。
- 润色旧文时，优先修正事实、结构、术语一致性、过长段落、标题表达和示例解释；不要无理由把整篇改成完全不同的文风。
- 扩写文章时，优先补“为什么这样做”“什么时候适用”“什么时候不适用”“怎么验证”“常见坑”；必要时补一张真正减少理解成本的图，而不是只为了篇幅而加水。
- 涉及仓库里已有的相近主题时，先搜索相关文章，避免重复铺陈同一套背景，必要时通过链接建立上下文。

## 交付习惯

- 如果用户只给了主题和零散材料，优先给出靠谱大纲、论点骨架或补充问题，然后再扩展成完整草稿。
- 如果用户要求直接落地到仓库，完成后自查 front matter、标题层级、代码块、资源引用、时间/版本前提和错别字。
- 对“写文章 / 起草 / 改写 / 润色 / 完善”这类任务，优先使用 repo-local skills：`article-writer` 与 `article-refiner`。
