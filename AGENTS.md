# blog-hugo agent guide

> **版本**: 1.2.0  
> **更新日期**: 2026-03-27  
> **适用范围**: Claude Code, KiloCode, Roo Code, OpenCode,  
> VS Code + AI 扩展

## 项目定位

- 这是一个 Hugo 技术博客仓库，主要内容源文件位于 `source/`，最终生成目录为 `public/`。
- 文章正文主要位于 `source/post/<year>/<id>.md`。
- `config.yaml` 当前启用了 `defaultContentLanguage: zh-cn`、
    `permalinks.post: /:year/:contentbasename`、`uglyURLs: true`。
- 这意味着文章的年份目录、数字文件名和 `id` 会直接影响最终链接；**除非用户明确要求，否则不要改已有文章的编号、年份目录或文件名**。

## 写作默认值

- 默认输出语言使用简体中文；仅在用户明确要求英文或双语时切换。
- 目标读者通常是有经验的软件工程师；优先写清问题、约束、权衡、落地方法，少空话、少营销腔。
- 开头前 1-3 段要交代：问题是什么、为什么值得读、读者读完能获得什么、本文不打算覆盖什么。
- 结论前置：每节优先给结论或适用条件，再解释原因，再给示例、代码或图。
- 长文优先使用清晰、描述性的 H2/H3 标题；让只扫标题的读者也能判断哪些段落和自己相关。
- 段落尽量短；流程、枚举、对比、决策条件优先用列表或表格表达，而不是堆成长段落。
- 代码示例前先说明它证明什么或解决什么；只展示和当前论点直接相关的片段。
- 如果某个结论依赖版本、平台、编译器、网络环境、时间点或实验条件，**必须显式写出前提**。
- **不要编造** benchmark、Issue 编号、官方结论、版本支持矩阵、引用来源或用户未提供的经历细节；不确定时明确写"待确认"或"需要验证"。

## Hugo / Markdown 约定

### Front Matter 规范

文章必须包含合法 YAML front matter，常见字段：

```yaml
---
title: 文章标题
date: 2025-03-27T10:00:00+08:00
draft: true              # 发布前改为 false
type: post
tags: [golang, performance]     # 克制、有检索价值
categories: [技术分享]          # 或 "最佳实践"、"故障排查"等
author: owent
---
```

**约束：**

- 修改已有文章时，优先保留现有 front matter、引用链接、脚注编号、资源文件名和整体章节骨架
- 不要擅自修改 `id`、文件名、年份目录

### 代码块语言标注

所有非 inline 的代码块（围栏代码块）**必须标注语言**，不允许使用无语言标注的裸代码块。纯文本或日志输出使用 ` ```text `。

### 图表使用指南

代码块、Mermaid、KaTeX、Draw.io、Excalidraw、Chart.js 仅在确实能降低理解成本时使用；不要为了"看起来更专业"而机械加图。

**如果纯文字会迫使读者脑内模拟**流程、状态机、调用时序、组件关系、跨区拓扑或数据变化，AI 应主动在对应段落附近补图，而不是等用户点名。

#### 图表选型规则

| 工具 | 适用场景 | 不推荐场景 |
|------|----------|------------|
| **Mermaid** | 流程图、时序图、状态图、类图、ER 图、甘特图、象限图、简单文本驱动架构图 | 需要复杂布局的架构图 |
| **Chart.js** | 有数值的比较、趋势、占比、分布、混合折线/柱状对比 | 系统结构图、概念关系图 |
| **Excalidraw** | 概念草图、粗粒度架构、角色/组件关系、需要"白板感"的解释型图示 | 需要精确几何关系的图 |
| **Draw.io** | 布局更规整、节点更多、需要稳定几何关系或更正式观感的架构/网络/部署图 | 简单流程、快速草图 |

#### 嵌入方式优先级

本仓库支持 2 种嵌入方式：代码块内嵌和图片语法引用外部文件。

**默认优先级：**

1. 简短图表：优先代码块（`mermaid`、`chart`/`chartjs`、`drawio`、`excalidraw`）
2. 较长或可复用的图表：优先拆成外部文件，用图片语法引用

> **不要**使用 Shortcode 方式（`{{< type src="..." >}}` 或
> `{{< type >}}...{{< /type >}}`）。虽然主题支持 Shortcode，但代码块和
> 图片语法已能覆盖所有场景，且更通用、更易维护。

#### 资源文件命名规范

新增配套资源时，请放在同一年目录下，并与文章编号保持同一前缀：

```text
source/post/2025/
├── 2601.md              # 文章正文
├── 2601-flow.mermaid    # 流程图
├── 2601-arch.excalidraw # 架构图
└── 2601-perf.chart.json # 性能图表
```

#### 图表说明要求

文件型图表的图片语法要写有意义的 alt 文本；正文中也要用 1-2 句说明"这张图要帮助读者看什么"，不要只丢一个图让读者自己悟。

**示例：**

```markdown
![MPG 调度模型](2601-mpg-model.mermaid)
*图 1：MPG 调度模型。M（Machine）是操作系统线程，P（Processor）是逻辑处理器，
G（Goroutine）是待执行任务。*
```

## 改文策略

- 写新文时，先补齐主题、目标读者、核心结论和章节结构，再写正文
- 润色旧文时，优先修正事实、结构、术语一致性、过长段落、标题表达和示例解释；不要无理由把整篇改成完全不同的文风
- 扩写文章时，优先补"为什么这样做""什么时候适用""什么时候不适用""怎么验证""常见坑"；必要时补一张真正减少理解成本的图，而不是只为了篇幅而加水
- 涉及仓库里已有的相近主题时，先搜索相关文章，避免重复铺陈同一套背景，必要时通过链接建立上下文

## 交付质量检查清单

交付前必须逐项确认以下检查项：

### Content Quality

- [ ] 标题具体、不标题党，能概括核心结论
- [ ] 摘要忠实反映正文，不夸大
- [ ] 开头 1-3 段说清问题、收益、边界
- [ ] 每节先给结论/takeaway，再解释原因
- [ ] 术语一致，前后统一
- [ ] 没有"正确的废话"（如"我们需要平衡性能与可用性"）

### Technical Accuracy

- [ ] 不编造 benchmark、Issue 编号、官方结论、版本支持矩阵
- [ ] 结论依赖版本/平台/编译器/时间点时，**必须显式写出前提**
- [ ] 不确定的信息标注"待确认"或"需要验证"
- [ ] 代码示例经过验证或明确标注"示意代码"

### Hugo/Markdown Compliance

- [ ] YAML front matter 合法，包含必要字段（title, date, type）
- [ ] **所有**围栏代码块都标注了语言（纯文本/日志用 `text`），无裸代码块
- [ ] 图表说明文字有意义（不只是"如图"）
- [ ] 资源文件命名符合 `<id>-*.ext` 格式
- [ ] 内部链接格式正确（`[文字](/year/id/)`）

### Readability

- [ ] 段落尽量短（3-4 行以内）
- [ ] 流程/枚举/对比优先用列表或表格
- [ ] H2/H3 标题描述性足够，只看标题能判断相关性
- [ ] 长段落是否可拆成 bullet points？

### Editorial

- [ ] 没有错别字或明显语法错误
- [ ] 中英文混排符合规范（中文与英文/数字间留空格）
- [ ] 标点符号使用正确

## 错误处理与边界情况

### 信息不足时

**不要**强行生成内容。应该：

1. 列出信息缺口清单
2. 提供可能的写作方向
3. 询问用户需要补充哪些信息

**示例回复：**

```markdown
根据现有材料，以下信息需要补充：
1. 目标读者是谁？（初学者/中级/专家）
2. 核心结论是什么？（读者应该获得什么）
3. 有代码示例或实验数据吗？
4. 适用场景和限制条件是什么？

在获得这些信息前，我可以提供：
- 文章大纲建议
- 可能的章节结构
- 相关参考资料（如有）
```

### 发现冲突时

如果主题与仓库已有文章有重叠：

1. 列出已有相关文章
2. 说明本文与已有文章的区别
3. 建议补充而非重复
4. 必要时在文章中添加链接

**示例：**

```markdown
## 与已有文章的关联

本文关于 MPG 调度器的讲解与 `/2023/1802/` 有重叠。
建议：
- 本文侧重实践应用，1802 侧重原理解析
- 本文简要提及原理，链接到 1802 获取详细说明
```

### 技术细节不确定时

**必须**显式标注，不要猜测：

```markdown
> **待确认**：该特性在 v1.18 中的行为可能与 v1.17 不同，需要验证。
```

### 发现已有文章错误时

如果是润色已有文章时发现错误：

1. 在修改说明中标注
2. 如果是事实错误，直接修正
3. 如果不确定，添加"待确认"标注
4. 如果是重大错误，建议用户单独确认

## 工具协作指南

### Skill 选择

| 任务类型 | 使用 Skill | 说明 |
|----------|------------|------|
| 写新文章 | article-writer | 从零创建内容 |
| 润色已有文章 | article-refiner | 修正、优化、完善 |
| 添加图表 | 两者均可 | writer 用于新文章，refiner 用于已有文章 |

### 多 Agent 兼容性

本仓库的 Skills 配置位于以下位置，保持同步：

| 目录 | 用途 | 状态 |
|------|------|------|
| `.agents/skills/` | 通用 Agent 目录（Agent Skills 规范），主副本 | ✅ 主源 |
| `.claude/skills/` | Claude Code 专用 | 同步自 `.agents/` |
| `.kilocode/skills/` | KiloCode 专用 | 同步自 `.agents/` |
| `.roo/skills/` | Roo Code 专用 | 同步自 `.agents/` |

**修改策略：**

1. 所有修改在 `.agents/skills/` 进行（含 `SKILL.md` 和 `references/` 子目录）
2. 运行 `sync-skills.ps1` 同步到其他目录
3. 确保 compatibility 字段涵盖所有支持的 Agent

### 配置文件关系

```text
AGENTS.md (本文件，全局规范的唯一真相)
├── 被 .claude/CLAUDE.md 引用
├── 被 .github/instructions/hugo-articles.instructions.md 引用
└── 被各 Skill 引用

.agents/skills/
├── article-writer/
│   ├── SKILL.md
│   └── references/embedding-syntax.md
└── article-refiner/
    ├── SKILL.md
    └── references/embedding-syntax.md
```

## 禁止事项

以下行为**严格禁止**：

1. **不要编造**任何未经验证的信息（benchmark、Issue 编号、官方结论、用户经历）
2. **不要擅自修改**已有文章的文件名、`id`、年份目录、稳定锚点
3. **不要**在信息不足时强行生成完整文章
4. **不要**为了"看起来更像 AI 在工作"而过度改写
5. **不要**删除作者有意保留的内容（即使你觉得冗余）
6. **不要**代替用户做技术选型决策（只提供建议）
